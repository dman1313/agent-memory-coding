# Symposium keynote: Rust, agents, and community toolchain plus tabular functional models

Date: 2026-05-06

## Meeting Report

### Session Context & Speaker Introductions

- Speaker 1 (Nico): lead designer of Rust since 2011; designed the borrow checker; senior principal engineer at Amazon; noted Rust usage across AWS services (S3, EC2, Lambda).
- Framing: Rust’s positioning as both low-level (kernel/system use) and high-level (applications, including TUI apps such as Ratatouille).
- Industry signals mentioned:
  - Linux kernel added Rust as a supported language alongside C.
  - Microsoft using Rust in kernel contexts.
  - Rust adoption accelerating alongside agentic development workflows.

### Rust + Agentic Development: Current Shift & Implications

- Two historic Rust adoption challenges described:
  - Large existing C/C++ codebases make rewrites expensive; interop introduces asymmetries.
  - Rust’s type system complexity increases learning curve and early drop-off.
- Shift described: agentic development changes priorities from “how fast can we write code” to “how maintainable is the resulting system,” because agents reduce manual coding effort while maintenance remains costly.
- Why Rust works well with agents (as presented):
  - Guardrails: type system guides agents toward correct solutions.
  - Versatility across domains (systems to apps).
  - Efficiency: enables fast development without sacrificing performance/cost efficiency at scale.
  - Error messages: rich compiler diagnostics reduce agent time spent searching context and accelerate automated fixes.

### Problem Statement: Stale Training Data & Out-of-Date Practices

- Example library: Toasty (released in April; aimed at making database work in Rust more accessible via macros and generated code).
- Demonstrated issues with agent behavior:
  - Claude did not recommend Toasty when asked for SQL + Rust libraries.
  - When prompted to use Toasty, it “searched the internet” and produced output using Toasty 0.3 while crates.io showed Toasty at 0.5.
  - Output also defaulted to Rust 2021 despite Rust 2024 being current, reflecting that much public Rust code uses older patterns.
- Core frustration stated: Rust/toolchain experts and crate authors have limited ability to influence how centralized agent vendors generate “best practice” Rust code.

### Symposium: Goals, Tooling Model, and Developer Workflow

- Symposium’s goal: an agent-targeted Rust toolchain enabling the Rust community (compiler contributors, library authors) to provide up-to-date guidance and crate-specific know-how directly to agents.
- Intended user workflow described:
  - Install Symposium via Cargo.
  - Initialize agent configuration via Cargo command to select agents.
  - Symposium configures hooks and supporting tooling behind the scenes.
- General guidance provided by Symposium (examples given):
  - Recommend Rust 2024 over Rust 2021.
  - Encourage `cargo add` to pull the latest crate version rather than relying on agent memory/training snapshots.
- “Skills” mechanism:
  - Skills are markdown files with metadata.
  - Skills can be declared for a specific crate (e.g., Toasty).
  - Symposium makes relevant skills available to the agent based on project dependencies, so the agent uses crate-author-provided guidance instead of scraping potentially outdated blog posts.

### Agent Accessibility Improvements for Rust Toolchains

- Human workflow baseline: Rust Analyzer lets humans jump into dependency source code; agent workflows typically lack this capability.
- Symposium enables agent-friendly alternatives (as described):
  - Provide the exact dependency source and version in use.
  - Support commands to fetch crate source information and allow inspection via standard tooling (grep/sed/awk-style workflows).
- Larger vision stated: an extensible ecosystem where many community participants contribute skills/extensions that improve agent outcomes.

### Interoperability & Standards for Agent Integrations

- Identified ecosystem problem: agent vendors use different conventions; even `agents.md` is not universally supported.
- Call to action stated:
  - Build interoperable standards so library authors can write guidance once and have it work across multiple agents (Codex, Claude, Qwen, etc.).
- Symposium’s bridging role:
  - Library authors provide extensions/services in a standard way.
  - Users choose agents.
  - Symposium converts/bridges between differing hook formats and conventions to reduce friction.

### Keynote: Open Compute / AI Compute for Agentic AI (Yong Hua Ling, BAAI)

- Scope adjustment: focus on AI compute/infrastructure; skip open evaluation due to 15-minute constraint.
- Thesis: agentic AI performance depends on the whole computation system (multi-runner execution, efficient inference), not just single-model quality; infrastructure is the next bottleneck.
- Context: rapid release of open-source/open-weight models enables:
  - On-prem deployment for security/privacy/control/resilience.
  - Post-training for innovation.
  - Need for more chipset options to lower cost and improve efficiency.
- DeepSeek v4 used as a stress test example for open compute:
  - Claimed suitability for agentic AI via 1M-token context length and strong reasoning/inference.
  - System-level optimizations described: hybrid attention, heterogeneous KVCache, FP4/FP8, new kernels, runtime optimization.
- Portability challenges highlighted:
  - Many deployed chipsets lack FP4/FP8 support.
  - Memory requirement: single device needs >80GB memory.
  - Operator kernels coupled to specific hardware backends.
- Open compute objective: decouple hardware-specific features via unified cross-chip software.

### FlatOS Stack & Related Components (BAAI)

- FlatOS described as an open-source system software stack supporting >30 chipsets (examples: Nvidia, Huawei, ARM, Sunrise 5).
- FlatOS packaged as skills to assist deployment, code generation, migration, and agent interaction.
- Compiler evolution:
  - Initially attempted a new Rust-based compiler ~3 years ago; vendor ecosystem not ready, so shifted to Triton.
  - Issues with mainstream Triton (as stated): limited backends; insufficient representation of hardware features; productivity-focused but less hardware-efficient.
  - FlatSheet: “augmented Triton compiler,” single codebase supporting 12 backends without vendor forks.
- Language evolution:
  - Triton Language Extension (TLE) introduced to balance productivity vs performance.
  - Three layers mentioned: Triton Language Lite, Structured, Raw.
  - Claimed strong performance gains vs native Triton and competitive results vs CUDA C / Ascend C.

### Kernel Porting, Operator Coverage, and Automation

- DeepSeek kernel situation described:
  - DeepSeek includes 67 CUDA kernels; needed replacement with Triton-based equivalents for cross-architecture support.
- FlatGems:
  - Claimed as the largest Triton operator library, with 400+ frequently used AI operators.
  - Integrated into PyTorch ecosystem.
  - Added a C++ wrapper to reduce Triton launch overhead.
- KernelGen automation platform:
  - Goal: generate and test thousands of kernels without manual expert implementation.
  - Claimed status: ~80% of kernels implemented via KernelGen.
  - Outputs: code generation, test cases, precision benchmarks, performance tests, iterative optimization loops.
  - Exposed via platform + MCP + skill packaging; claimed fewer tokens than general “cloudy code” while achieving better performance.
  - For DeepSeek: at least three new kernels implemented via this tool; enabled rapid migration in 1–2 days.
  - Supports Triton TLE and relies on agents rather than humans learning new syntax.

### Release Pipeline, Model Distribution, and Engine Integration

- FlatRelease pipeline:
  - Downloads open-source model code, migrates to target hardware, tests, and re-releases.
- Distribution:
  - Releases made on Hugging Face for multiple models (Minimax, GLM, Qwen, MiniCPM, Llama, others).
  - Delivered artifacts: migrated code and Docker images; “three steps” to run described.
- DeepSeek v4 outcome claimed:
  - Migrated including “Prop version and Flash version.”
  - Supported on 10 different AI chipsets.
  - Toolchain and precision conversion resources provided via a one-stop GitHub repository.
- Inference/training engine integration:
  - Identified gap: vLLM and SGLang lack strong multi-backend support.
  - Built a unified plugin system to integrate FlatOS into vLLM, SGLang, Megatron-LM (training), TE (training), and VRL (reinforcement learning).
  - Next planned plugin: SGLang within the month (as stated).

### Keynote: Tabular Functional Models & Structured Data as AI Frontier (Alex, Fundamental)

- Speaker background:
  - Head of Applied AI at Fundamental; role described as bridging science and enterprise applications.
  - Personal notes shared: lived in six countries (France, Switzerland, New York, Dubai, Hong Kong, Singapore); relocating to San Francisco; proposed to Korean girlfriend; requested audience photo proof.
- Motivation arc:
  - 2011 inspiration: Moneyball (analytics reducing bias) and HBR “Data Scientist” framing.
  - 2011–2012: early scikit-learn era cited as formative for the ML ecosystem.
  - 2022: GPT-3.5 prompted concern that traditional data science might become less relevant; reflected on skill transferability.
- Core claim: most enterprise information and decisions are structured (tables/Excel/CRM/databases; binary/numeric decisions like loan approval, hospital readmission).
- Limitation noted: LLMs are next-word predictors and are strong at unstructured text, but weaker at precise numerical reasoning; example referenced of arithmetic error (5.8 − 5.11).
- Status quo described: structured-data AI is still dominated by classical ML workflows (cleaning, feature engineering, model training such as random forest/XGBoost) repeated per use case; process largely unchanged for ~20 years.
- Tabular Functional Models (TFMs) concept:
  - Pretrain a model over many tables (millions) to learn priors about how tabular data is generated and what features mean (e.g., zip codes, square footage).
  - Hypothesis/claim: a pretrained tabular model can be more accurate and easier to use than traditional ML equivalents.
  - Claimed research trend: TFMs now outperform traditional ML approaches; described as a “GPT-3.5 moment” for tabular data.
  - Referenced Kyle B. (Kyle Barocco) as a key contributor; cited TabiQL as an open-source tabular functional model from INRIA.
- Fundamental positioning:
  - Product/model name: Nexus (their tabular functional model).
  - White paper available via company website.
  - Demo opportunity: lunch-and-learn at 12:00 PM to demonstrate the model.
  - Company details shared: raised $255M at $1.4B valuation; team with European roots; science team based in Barcelona; founding advisor Kyle Barocco (also tied to scikit-learn and INRIA).
- Closing framing:
  - “Right brain” = LLMs (creative, unstructured generation).
  - “Left brain” = tabular models (reasoning/mathematical, structured prediction engine for enterprises).
